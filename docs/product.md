# 小动物餐厅玩法文档

## 1. 产品定位

《小动物餐厅》是一款移动端优先的轻度跨端餐厅经营小游戏。

短期目标是 Web 版本，长期目标是微信小游戏和抖音小游戏。Web 版本也必须由 Cocos 工程编译产出；Web、微信小游戏、抖音小游戏共用同一套 Cocos 客户端代码。游戏由 Cocos 前端承载可交互营业过程，由 Node.js 后端保存玩家进度并负责核心经济结算。

## 2. MVP 核心循环

核心循环：

```text
消耗体力开始营业
  -> 90 秒可交互经营
  -> 后端结算金币
  -> 自由升级一个餐厅部件
  -> 下一次营业期望收入提升
  -> 五个部件满星
  -> 整体升级餐厅
  -> 进入下一轮部件升级
```

主循环目标：

- 玩家每次正常完成一局营业，期望刚好获得升级任意一个餐厅部件所需的金币。
- 任意部件升 1 星后，下一次营业的期望总收入固定提升 8%。
- 玩家可以自由选择升级部件，不提供系统推荐升级提示。
- 升级不同部件影响经营手感，但经济收益统一由全局成长公式控制。

## 3. 营业玩法

一次营业是可交互经营过程，不是即时结算。

规则：

- 每次营业固定消耗 10 体力。
- 标准营业时长为 90 秒。
- 营业界面提供免费常驻的 2 倍速入口。
- 2 倍速只缩短现实时间，游戏内仍按 90 秒推进。
- 同样表现下，1 倍速和 2 倍速最终收益一致。
- 营业结束后由后端根据过程摘要计算金币。

MVP 操作采用点击为主的三步服务链：

1. 顾客进店后，玩家点击空餐桌安排入座。
2. 顾客出现点餐/上菜状态后，玩家点击顾客或餐桌完成上菜。
3. 顾客用餐完成后，玩家点击收银机或顾客完成收银。

暂不做拖拽菜品、服务员寻路、厨房制作小游戏、多菜谱选择。

## 4. 营业密度

首版 90 秒营业的目标服务密度：

- 初始餐厅同时可接待 2 位顾客。
- 首版场景预留 5 个桌位；未解锁桌位显示锁定贴图且不可点击。
- 每局开场先进入 2 位初始顾客，保证玩家立即开始操作。
- 等待队列最多显示 4 位顾客，超过后暂停自然进店，等空位或顾客离开后继续。
- 顾客平均每 6 到 8 秒进入一位。
- 单个顾客完整流程约 18 到 25 秒。
- 普通玩家每局完成约 10 到 12 位顾客。
- 表现优秀玩家每局完成约 13 到 14 位顾客。
- 后端校验上限可先设为 18 位顾客，超过则判定为异常或截断。
- 客户端自然进店也必须遵守每局 18 位顾客上限，避免正常游玩产生后端不可结算的摘要。

顾客等待规则：

- 顾客有耐心值。
- 等待太久会离开。
- 餐桌状态需要显示阶段剩余时间或耐心倒计时，让玩家能判断上菜/收银优先级。
- 离开的顾客计入 `customersLost`。
- 顾客离开只降低表现评分，不倒扣金币。
- 已完成顾客的收益不受倒扣影响。

## 5. 顾客与主题

MVP 中，小动物先作为顾客外观和任务主题，不做种族玩法差异。

规则：

- 顾客类型先只有 `normal`。
- 可展示不同动物外观，例如猫、狗、兔子、熊。
- 不同外观在 MVP 中不改变耐心、金币、用餐时长等数值。
- 数据结构预留 `customerType`，后续可扩展特殊顾客。

后续可扩展方向：

- 急性子顾客。
- 高消费顾客。
- 慢食顾客。
- VIP 顾客。
- 特殊动物顾客。

## 6. 体力系统

体力配置：

```text
staminaMax = 60
staminaRecoverInterval = 5 分钟恢复 1 点
sessionStaminaCost = 10
```

节奏含义：

- 满体力可以连续营业 6 次。
- 1 倍速连续玩约 9 分钟。
- 2 倍速连续玩约 4.5 分钟。
- 消耗一局所需 10 体力后，需要约 50 分钟自然恢复。
- 从 0 回满 60 体力需要约 5 小时。

体力恢复由后端时间计算，避免本地改时间刷体力。
前端展示的下一点体力倒计时和满体力时间也应使用后端返回的恢复状态，不用本地自行推算。

## 7. 经济模型

### 7.1 核心目标

任意部件升 1 星后，下一次营业的期望总收入固定提升 8%。

经济收益和玩法手感分离：

- 经济收益：任意部件升 1 星，都让 `incomePower + 1`。
- 玩法手感：不同部件影响不同经营体验，但不额外叠加隐藏收入倍率。

### 7.2 基础公式

初始值：

```text
baseRevenue = 100
restaurantLevel = 1
每个部件星级 = 0
```

全局收入成长：

```text
incomePower =
  (restaurantLevel - 1) * 25
  + cashierStar
  + tableStar
  + chairStar
  + floorStar
  + wallStar

expectedRevenue = round(baseRevenue * 1.08 ^ incomePower)
upgradeCost = expectedRevenue
```

说明：

- 每个餐厅等级内有 5 个部件。
- 每个部件可从 0 星升到 5 星。
- 每个餐厅等级内共有 25 次部件升级。
- 同一 `incomePower` 下，所有未满星部件的下一星升级成本相同。
- 玩家选择升级哪个部件只影响经营手感，不影响经济效率。

示例：

```text
incomePower 0: 期望收入 100，升级成本 100
incomePower 1: 期望收入 108，升级成本 108
incomePower 2: 期望收入 117，升级成本 117
incomePower 3: 期望收入 126，升级成本 126
incomePower 4: 期望收入 136，升级成本 136
incomePower 5: 期望收入 147，升级成本 147
```

金币均使用整数，公式计算后四舍五入。

### 7.3 营业表现系数

实际收入：

```text
actualRevenue = round(expectedRevenue * performanceFactor)
```

表现系数：

```text
performanceFactor = clamp(
  0.75
  + completionScore * 0.35
  + satisfactionScore * 0.15
  + comboScore * 0.05,
  0.75,
  1.30
)
```

指标说明：

- `completionScore`: 0 到 1，本局成功完成服务/收银的顾客比例。
- `satisfactionScore`: 0 到 1，顾客等待少、没有超时离开则更高。
- `comboScore`: 0 到 1，连续及时完成服务获得少量奖励。

体验结果：

- 很差表现：约 0.75 倍，可能不足以升级一个部件。
- 普通表现：约 1.00 倍，刚好升级一个部件。
- 优秀表现：约 1.15 到 1.30 倍，升级后留少量余钱。

正常完成营业时，保底收入不低于 `expectedRevenue * 0.75`。

## 8. 餐厅部件升级

五个可升级部件：

- 收银机。
- 餐桌。
- 餐椅。
- 地板。
- 墙面。

升级规则：

- 玩家自由选择任意未满星部件升级。
- 不提供推荐升级提示。
- 每个部件每个餐厅等级内从 0 星升到 5 星。
- 同一成长进度下，所有未满星部件升级成本相同。
- 部件满 5 星后禁用升级按钮，等待整体升级餐厅。

升级界面必须展示：

- 当前星级。
- 升级成本。
- 下一星效果。
- 金币不足时差多少金币。
- 满星状态。

## 9. 部件手感效果

每次升级任意部件都会带来统一经济收益：`incomePower + 1`。

不同部件同时提供不同经营手感：

- 收银机：收银点击窗口更宽，或收银动画更快。
- 餐桌：桌位/接待表现更清晰，关键星级可增加同时可坐顾客数。
- 餐椅：顾客耐心更高，满意度更容易维持。
- 地板：顾客移动更快，翻台更顺。
- 墙面：顾客进入节奏更平滑，空窗期更少。

这些手感变化可以帮助玩家更容易打出高完成率、满意度和连击，但后端经济基准只由 `incomePower` 控制。

## 10. 整体升级餐厅

当五个部件都达到 5 星时，解锁整体升级餐厅。

整体升级餐厅效果：

- `restaurantLevel + 1`。
- 五个部件星级重置为 0。
- 餐厅进入下一视觉阶段或主题阶段。
- 下一轮部件成本和营业收入按新的 `incomePower` 计算。
- 不直接发体力、金币、道具等奖励。

奖励放入任务系统处理，而不是由整体升级本身直接发放。

体验原则：

- 部件星级 UI 会在新餐厅等级内重置。
- 用户玩法体验不能回退。
- 实际能力采用“餐厅等级基础值 + 当前部件星级增量”的模型。
- 升级餐厅后，基础客流、基础耐心、基础移动速度、基础收银效率等长期体验成长应被保留。

`incomePower` 在整体升级前后不下降：

```text
餐厅 1 级，五部件全 5 星:
incomePower = (1 - 1) * 25 + 25 = 25

升级到餐厅 2 级，五部件重置为 0 星:
incomePower = (2 - 1) * 25 + 0 = 25
```

## 11. 任务系统

MVP 包含轻量任务系统。

任务系统定位：

- 辅助推进，不替代主循环。
- 总奖励强度控制在主循环收益的 20% 到 30%。
- 不改变 `incomePower`。
- 发放金币、体力，后续可扩展道具或装饰。

MVP 任务类型：

任务界面必须明确区分引导任务、每日任务、成长任务，避免玩家把一次性成长目标和每日刷新目标混在一起。

### 11.1 引导任务

- 完成首次营业。
- 完成首次部件升级。
- 完成首次任务奖励领取。
- 完成首次整体升级餐厅。

### 11.2 每日任务

- 完成 3 次营业。
- 升级 3 次部件。
- 获得一定数量金币。
- 服务一定数量小动物顾客。

每日任务奖励：

- 每日任务总金币奖励约等于 1 到 2 次正常营业收入。
- 每日任务总体力奖励约 10 到 20 点。
- 每日任务进度按后端日期刷新，每个每日任务每天最多领取一次，次日重新完成后可以再次领取。

### 11.3 成长任务

- 餐厅升到 2 级。
- 任意部件首次升到 5 星。
- 五个部件首次全部升到 5 星。
- 累计服务一定数量顾客。
- 累计完成一定次数营业。

整体升级餐厅相关奖励放在成长任务里，例如“餐厅升到 2 级”任务。

## 12. 新手体验

初始玩家状态：

```text
coins = 0
stamina = 60
restaurantLevel = 1
cashierStar = 0
tableStar = 0
chairStar = 0
floorStar = 0
wallStar = 0
incomePower = 0
```

首轮体验：

1. 玩家进入餐厅主界面，看到金币 0、体力 60。
2. 高亮“开始营业”。
3. 首次营业中用轻量提示引导三步操作：入座、上菜、收银。
4. 首次营业结束后，获得约 100 金币。
5. 高亮升级入口。
6. 玩家自由选择任意一个部件升 1 星。
7. 通过引导任务领取少量奖励。

不做长强制教程，采用“轻量引导任务 + 首次操作高亮”。

## 13. MVP 界面范围

首版只做 4 个核心界面：

1. 餐厅主界面：展示金币、体力、餐厅等级、五个部件星级、开始营业按钮。
2. 营业界面：90 秒经营关卡，包含顾客、餐桌、收银机、倒计时、连击/满意度反馈、2 倍速入口。
3. 升级界面：五个部件自由升级，展示当前星级、升级成本、下一星效果；不显示推荐提示。
4. 任务界面：每日任务、成长任务、引导任务，领取金币/体力奖励。

移动端 UI 原则：

- 竖屏优先。
- 适合触摸点击。
- 避免关键按钮贴边。
- 适配安全区、刘海、底部手势区域。
- Web 预览也必须检查移动端视口。

## 14. 后端权威与结算

核心进度由后端保存：

- 金币。
- 体力。
- 体力恢复时间。
- 餐厅等级。
- 五个部件星级。
- 营业记录。
- 任务系统进度。

前端负责：

- 营业过程表现。
- 动画和操作反馈。
- 本局临时分数和过程摘要。

营业开始：

- 前端请求后端开始营业。
- 后端校验体力是否至少 10。
- 后端扣除 10 体力。
- 后端创建营业会话。

营业结束：

- 前端提交营业过程摘要。
- 后端读取玩家餐厅等级和部件星级。
- 后端计算 `expectedRevenue`、`performanceFactor` 和最终金币。
- 后端写入金币变化和营业记录。

后端不信任前端直接提交的金币数。

## 15. 中断处理

营业会话规则：

- 开始营业时立即扣 10 体力。
- 正常 90 秒完成后按过程摘要结算。
- 玩家中途退出、断线、切后台时，允许在短时间内恢复本局。
- MVP 恢复窗口建议为 2 分钟。
- 超过恢复窗口后，后端按最低保底或已完成摘要结算。
- 同一个营业会话只能结算一次。
- 后端设置会话过期时间，防止无限挂起营业。

## 16. 登录与身份

MVP Web 开发期可以使用匿名玩家 ID 或测试 token。

长期方向：

- 微信小游戏接入微信身份。
- 抖音小游戏接入抖音身份。
- 后端仍保持玩家进度权威存档。

账号体系不阻塞首版玩法原型。

## 17. MVP 不做范围

MVP 不做：

- 离线收益。
- 商城。
- 背包。
- 装饰换装。
- 排行榜。
- 好友系统。
- 广告商业化。
- 支付。
- 复杂菜谱。
- 厨房制作小游戏。
- 服务员寻路 AI。
- 多种顾客数值差异。
- 长强制教程。

这些系统可以后续作为独立任务设计。

## 18. 美术与体验方向

剩余简单边界按以下方向闭环：

- 主题：温暖、可爱、轻松的小动物餐厅。
- 视觉优先级：先保证核心经营物件清晰可点，再追求装饰丰富度。
- 小动物表现：可爱、易识别、状态反馈清楚。
- 营业反馈：服务成功、顾客离开、连击、收银成功都要有明确反馈。
- UI 风格：移动端优先，按钮清晰，信息密度适中。
